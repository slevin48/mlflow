   335→### Complementarity Assessment
   336→
   337→This is the core deliverable analyzing how well skore, MLflow, and JupyterLab work together as an integrated pre-MLOps experimentation workflow.
   338→
   339→#### 1. Workflow Coherence: Does the skore -> MLflow handoff feel invisible or jarring?
   340→
   341→**Jarring.** The handoff happens at the data layer (skore logs metrics, params, tags, and artifacts into MLflow automatically), which works seamlessly. But at the user experience layer, the handoff is invisible in the worst sense: there is no acknowledgment, no link, no inline preview. After `project.put()` in a notebook, the user gets a print statement and nothing else. They must manually:
   342→1. Open a new browser tab
   343→2. Type `localhost:5000`
   344→3. Scroll past the GenAI landing page
   345→4. Find their experiment in "Recent Experiments"
   346→5. Click through to Training Runs
   347→6. Click into the specific run
   348→
   349→This 6-step manual navigation after logging data makes the handoff feel like two completely separate tools rather than an integrated workflow.
   350→
   351→#### 2. Mental Model Gaps: What do practitioners expect that does not exist?
   352→
   353→| Expectation | Reality |
   354→|-------------|---------|
   355→| Landing page shows my experiments | Landing page promotes GenAI features |
   356→| Clicking an experiment shows runs | Clicking shows empty observability charts |
   357→| Cross-validation folds visible as expandable rows | Folds hidden, count ambiguous |
   358→| Side-by-side fold comparison table | Each fold on separate page |
   359→| Download button on artifacts | No download button visible |
   360→| Clickable link from notebook to MLflow | No link provided |
   361→| Autocomplete for skore API | No completions available |
   362→| Inline preview of logged data in notebook | No inline feedback |
   363→| Register Model button on run page | Button missing |
   364→
   365→#### 3. Context Switching Cost
   366→
   367→A typical experiment with skore+MLflow+JupyterLab requires:
   368→
   369→- **3 browser tabs/windows:** JupyterLab, MLflow UI, documentation
   370→- **1 terminal:** MLflow tracking server (`mlflow ui`)
   371→- **Total context switches per experiment cycle:** 4-6 (write code -> run cell -> switch to MLflow -> navigate to run -> switch back -> iterate)
   372→
   373→For comparison, W&B requires 2 tabs (notebook + dashboard) with inline links. The current workflow has approximately 2x the context switching cost of comparable tools.
   374→
   375→#### 4. Discovery Gap: How easily can a new user understand that skore logs to MLflow automatically?
   376→
   377→**Not easily at all.** There is no indication anywhere in the MLflow UI that skore logged the data. The tags (`skore_status`, `skore_version`, `report_type`) are present but not surfaced prominently. A new user seeing the MLflow experiment would not know:
   378→- That skore was involved
   379→- That the rich artifacts (confusion matrix, ROC curve, HTML analysis) came from skore's automatic logging
   380→- That `report.pkl` can be loaded back with skore
   381→- That the `_std` metric variants represent cross-validation standard deviations
   382→
   383→The discovery path is: read the skore documentation -> understand mode="mlflow" -> know to look in MLflow. There is no reverse discovery path (MLflow UI -> understanding skore was used).
   384→
   385→#### 5. Round-Trip Fidelity: Can users retrieve and inspect stored reports without reading docs?
   386→
   387→**Partially.** Users can view:
   388→- Metrics (accuracy, precision, recall, etc.) directly in the MLflow run detail
   389→- Artifact PNGs (confusion matrix, ROC, precision-recall) via inline preview
   390→- HTML analysis report via inline preview
   391→- CSV metrics file via inline preview
   392→
   393→Users **cannot** easily:
   394→- Download any of these files (no download button)
   395→- Load the `report.pkl` back without knowing skore's API
   396→- Compare metrics across folds without visiting each child run individually
   397→- Export run data for use in their notebook
   398→
   399→The visual inspection is good; the programmatic round-trip requires documentation.
   400→

   417→The data foundation is strong (skore logs comprehensive metadata), but the UI workflow for promotion review is incomplete. An MLOps engineer would need to fall back to the REST API or Python client for actual promotion decisions.
